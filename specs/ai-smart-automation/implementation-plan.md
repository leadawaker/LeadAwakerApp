## Phase 1: AI-Powered Contextual Bumps

### 1.1 Schema update
- **File:** `shared/schema.ts`
- Add `isAiGenerated` boolean field to Interactions table
- Add `aiBumpEnabled` boolean field to Campaigns table (opt-in per campaign, default false)

### 1.2 Bump scheduler AI path
- **File:** `automations/src/automations/bump_scheduler.py`
- Before selecting template, check `lead.conversion_status`
- If status in (`Multiple Responses`, `Qualified`, `Call Booked`) AND campaign has `ai_bump_enabled`:
  - Load conversation history via `get_interactions_for_lead(lead_id, limit=15)`
  - Load campaign context (agent_name, ai_style, service, usp, business_description)
  - Call new function `generate_ai_bump(history, campaign_context, bump_stage)`
  - Validate response through output guardrails
  - On success: send AI message, mark `is_ai_generated = true`
  - On failure: fall back to template bump

### 1.3 AI bump generator
- **New file:** `automations/src/automations/ai_bump_generator.py`
- System prompt: persona + "You are following up with a lead who went quiet. Reference the previous conversation naturally. Keep it under 160 chars for SMS or 300 for WhatsApp. Sound human, not salesy."
- User prompt: conversation history + bump context (days since last message, bump stage)
- Provider: Groq (fast, cheap) or Claude depending on quality needs
- Timeout: 30s, fallback to template

### 1.4 Campaign UI toggle
- **File:** `client/src/features/campaigns/components/CampaignDetailView.tsx`
- Add "AI-Powered Bumps" toggle in the AI Settings section
- Tooltip: "When enabled, bump messages for engaged leads are generated by AI based on conversation history instead of using templates."

## Phase 2: Buying Signal Detection

### 2.1 Signal classifier
- **New file:** `automations/src/automations/buying_signal_classifier.py`
- Stage 1: Regex pre-filter for obvious patterns ("call me", "ready to buy", "sign up", "how much", "pricing")
- Stage 2: Groq classifier for nuanced signals (context-dependent phrases)
- Returns: `{ detected: bool, signal_type: 'immediate' | 'scheduling' | 'price_ready', confidence: float }`

### 2.2 Integration into inbound handler
- **File:** `automations/src/automations/inbound_handler.py`
- After guardrail checks, before AI conversation:
  - Check if campaign `booking_mode_override = 'Call Agent'`
  - If yes, run buying signal classifier on inbound message
  - If signal detected: set flag on lead, notify closer, modify AI response behavior

### 2.3 AI response override for buying signals
- **File:** `automations/src/automations/ai_conversation.py`
- When buying signal flag is set, inject into AI prompt:
  - "The lead just expressed buying intent. Respond as if you're the business owner checking your schedule. Say something like: 'Let me check my agenda, I'll call you right back.'"
- Configurable response template per campaign (new field: `buyingSignalResponse`)

### 2.4 Closer notification
- **File:** `automations/src/automations/inbound_handler.py`
- New function `notify_buying_signal(lead_id, account_id, signal_type, message)`
- Channels: Telegram + in-app + web push
- Content: lead name, phone, exact message, last 3 messages for context
- Link: `/conversations?lead={lead_id}`

### 2.5 Fallback timer
- **File:** `automations/src/scheduler/jobs.py`
- New job: `buying_signal_followup` runs every 5 minutes
- Checks leads with `buying_signal_detected_at` > 15 min ago AND no outbound message since
- AI generates fallback: "Can't do it right now, let me find another time for you"
- Clears the signal flag after fallback

### 2.6 Schema changes
- **File:** `shared/schema.ts`
- Leads table: `buyingSignalDetectedAt` (timestamp), `buyingSignalType` (text)
- Campaigns table: `buyingSignalResponse` (text, configurable template)

- **File:** `shared/schema.ts`

  id: serial primary key
  accountId: integer (FK to Accounts)
  category: text (pricing, services, faq, team, hours, location, policies, testimonials)
  title: text
  content: text
  updatedAt: timestamp
  createdAt: timestamp

  - Triggers `manual_takeover = true`
  - Tags: `knowledge_gap_handoff`

## Phase 4: Smart Human Handoff

### 4.1 AI detection handling
- **File:** `automations/src/automations/ai_conversation.py`
- Track `ai_detection_count` in conversation metadata (or lead field)
- Count 1: AI deflects naturally (persona-appropriate)
- Count 2: AI admits being AI, offers to still help
- Count 3+: Handoff triggered

### 4.2 Circular conversation detection
- **File:** `automations/src/automations/ai_conversation.py`
- After generating response, check if the last 3 AI responses address the same topic
- Use Groq classifier: "Are these 3 responses about the same unresolved question? yes/no"
- If yes: handoff with reason "conversation_loop"

### 4.3 Handoff reason tracking
- **File:** `shared/schema.ts`
- Leads table: `handoffReason` (text: guardrail, buying_signal, ai_detected, knowledge_gap, conversation_loop, explicit_request)
- This is metadata alongside `manualTakeover` boolean

### 4.4 Unified handoff function
- **File:** `automations/src/automations/inbound_handler.py`
- Refactor `notify_manual_takeover` to accept a reason parameter
- Include reason in notification: "Lead requested human contact" vs "AI couldn't answer a pricing question"
