# Requirements: Gemini Live Voice (voice-to-voice memo replies)

## What & Why

When a lead sends a WhatsApp voice memo and the campaign replies in voice, the reply today goes through a three-model pipeline: Groq Whisper transcribes the memo → the campaign's chat model (GPT/Groq) writes a text reply → Gemini TTS speaks it. The transcript loses tonality: hesitation, skepticism, warmth, irony never reach the AI.

This feature routes **voice turns** through the Gemini Live API instead: the model hears the lead's actual audio (understanding tone directly) and produces the spoken reply in one step. Latency is irrelevant (voice memos are asynchronous), so each turn opens a short-lived Live session, exchanges one turn, and closes.

Decisions locked in conversation (2026-07-03):

- **Native audio is the default** engine for opted-in campaigns; half-cascade available as a setting (`voice_engine`: `pipeline` | `live_native` | `live_half`). `pipeline` (current behavior) stays the global default — campaigns opt in.
- **Split-brain accepted:** text turns stay on the campaign's existing chat model; only voice turns use Gemini Live. No porting of text turns.
- The Live system instruction is the **same rendered campaign prompt** (prompt 93 flow via `build_conversation_prompt` with `voice_mode=on`); its voice-mode section was already rewritten tag-free for Gemini (2026-07-03).
- Scheduled bumps stay on plain Gemini TTS (no conversation to react to). Voice mismatch between native-audio replies and TTS bumps is accepted.

## Acceptance Criteria

1. A campaign with `voice_engine = live_native`, on a WhatsApp channel with `voice_reply_mode = voice_reply`: lead sends a voice memo → reply arrives as a voice memo generated by Gemini Live native audio, in the account's configured voice, in the campaign language.
2. The reply's transcript is stored as the interaction's Content — the CRM inbox, buying-signal detection, smart handoff, sentiment, and nightly summaries all keep working unchanged.
3. The lead's actual memo audio (not just the transcript) is fed to Live when available; for history or missing audio, transcripts are used.
4. Any Live failure (quota, timeout, model error) falls back to the existing pipeline (Whisper transcript → chat model → Gemini TTS) in the same turn — the lead always gets a reply.
5. `voice_engine = live_half` uses the half-cascade model (same TTS voices as bumps) with identical plumbing.
6. `voice_engine = pipeline` (default) leaves current behavior byte-for-byte unchanged.
7. Campaign settings UI exposes the engine choice (en + nl), auto-saving like other campaign settings fields.

## Dependencies / Related

- Gemini TTS migration (built 2026-07-03) — shares `GEMINI_API_KEY`, OGG/Opus conversion helper, voice-name columns.
- `voice_reply_mode` per campaign/account (existing) still decides *whether* a turn is voice; `voice_engine` decides *how* it's synthesized.
- Future: the bump-3 outbound calling feature will reuse the Live session module built here (real-time instead of turn-based).
- Out of scope: tone-annotated transcription for the pipeline path; group/multi-memo batching changes; PT language.
