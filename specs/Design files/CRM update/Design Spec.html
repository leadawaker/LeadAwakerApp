Lead Awaker — Design System

One warm room,
lit from the side.

Every surface in Lead Awaker is paper resting on a warm bone ground, raised or recessed by a single light source in the upper-right. This document is the source of truth for that language — tokens, type, components, and the conventions that keep the build coherent across desktop and mobile.

01 — Foundations

Brand & the light model

The aesthetic is warm-bone neumorphism: soft, tactile surfaces that feel pressed into or lifted out of the page, finished with a single editorial wine accent. Nothing is flat; nothing is loud.

Principles
Directional light
A single warm light sits upper-right (--lx .94 / --ly -.34). Every raised and inset shadow derives from it, so elevation reads consistently. Never hand-author a shadow — use the tokens.
Paper on bone
Content lives on white --card paper floating over the warm --bg ground. Chrome (nav, rails) sits on the slightly deeper --bg-2 to frame the bright content.
Wine, used sparingly
Wine is the only brand color. It marks the active state, the primary action, and the brand dot — never decoration. If everything is wine, nothing is.

02 — Library

Buttons

Three tiers. One wine primary per view, neumorphic neu for secondary actions, and quiet ghost for tertiary. Primary and secondary use mono uppercase labels; ghost stays sentence-case sans.

Primary — .btn-wine
The single most important action. Wine gradient, lifts on hover, presses inset on click.
Secondary — .btn-neu
Raised paper. Most actionable buttons across the app.
Tertiary — .btn-ghost
Outlined / icon-only. Dismissals, toolbar utilities.
Compact actions — .la-btn
The mono-uppercase actions that fill toolbars, detail headers, send/handoff and FABs. Base .la-btn + one surface modifier (--wine / --soft / --inset) + optional --icon / --lg / --pill. This is the family the build must route through — the wine gradient lives in --wine-grad, set once.

Messaging

The lead conversation reads as two distinct voices. Our AI agent speaks in solid ink balloons with light cream text — deliberately flat, no neumorphism, so automated replies are unmistakable. The lead's incoming messages are inset wells pressed into the bone ground, the same carved affordance as a field.

03 — System

UX patterns

Locked interaction decisions that apply across the product. These are settled — build to them rather than re-exploring.

A

List → detail drills open as a bottom sheet (mobile)

Every drill-down rises as a dimmed sheet with a drag handle (MobSheet); the underlying screen scales back via MobRecede. Push and expand transitions were explored and dropped.

B

Selection lifts, rest stays flat

In any list, only the selected row carries elevation + a wine left-bar (.la-camp-card.active). Unselected rows are flat with inset glyph tiles, so the eye lands on one place.

C

Calendar defaults to agenda on phone

The month grid is unusable at phone width — the day/agenda list is the mobile default, with month offered only as a secondary toggle.

D

One unified mobile app, not responsive pages

The bottom tab bar is the navigation; all screens live in one host and switch via the router. Touch targets never drop below 44px.

03 — System

Responsive behavior

Desktop chrome reflows by viewport width; the Leads detail panel reflows by its own container width (via ResizeObserver), so it adopts the rich layout on a wide monitor and stacks when squeezed. Below the phone breakpoint the app hands off entirely to the mobile host. Breakpoint literals mirror the --bp-* reference tokens.

Range Width Key changes
Mobile< 768Hands off to the mobile host. Bottom tab bar (--bottombar-h), drill-downs as sheets, calendar defaults to agenda, decorative blobs hidden.
Tablet768–1023Sidebar persists. Detail panel stacks single-column; tables and kanban scroll horizontally rather than compress.
Desktop1024–1239In Leads list view the collection pane collapses to a 36px icon rail with hover-card peek (< --bp-desktop), freeing room for the detail panel.
Desktop-L1240–1679Full list + detail. Detail adopts the wide row layout once its container clears ~780px (contact / chat / score side by side).
Wide≥ 1680Detail container clears ~1180px and the chat + AI summary split into their own columns; kanban shows all stages without scroll.
Collapsing strategy
·

Container queries for the detail panel, viewport for chrome

The Leads detail reflows on its measured width (780 / 1180), not the window — so table & pipeline views, whose detail is a narrower side panel, stack correctly even on a wide screen.

·

List collapses to an icon rail before it cramps

Below --bp-desktop (1240) the Leads list becomes a 36px avatar rail with a hover card, rather than a squeezed two-line row.

·

Wide data scrolls, never compresses

Tables keep fixed column widths and scroll-x past TABLE_MIN; kanban columns hold 256px and scroll-x. Don't shrink cells to fit.

·

Decoration is the first thing to go

Floating blobs and ambient light flourishes are desktop-only; they're hidden at mobile so content centers cleanly. Touch targets never drop below 44px.

03 — System

Code conventions

How the system stays coherent as it grows. These rules come straight from the build and prevent the failure modes we've already hit.

1

Token-only color & depth

Never hardcode a hex or hand-author a shadow. Reference --ink, --wine, --sh-raised-medium, etc. This is what lets the whole UI re-light from one set of variables.

2

Prefix shared CSS with la-

App-level component classes (.la-nav-item, .la-status, .la-mono-tile) carry the la- namespace and live in design-system.css — not in page files.

3

Per-tab name prefixes in JSX

Each Babel script shares one global scope via Object.assign(window, …), so two files defining StatusPill clobber each other. Prefix a tab's local helpers: Tasks → MT*, Leads → ML*, Calendar → MC*.

4

Name every styles object uniquely

Never const styles = {…} in a shared script — collisions break the build. Use const billingStyles = {…} or inline styles.

5

Port logic, don't load whole view files

To reuse a desktop tab on mobile, load only its *-data.js and re-implement the needed bits with prefixed names. Don't load tasks-views.jsx wholesale into the mobile host.

Do
  • Use one wine primary action per view
  • Set fields inset, buttons raised
  • Lead state pills with a colored dot
  • Reach for a surface class before restyling
  • Keep eyebrows mono, uppercase, 0.2em
Don't
  • Invent new hexes or gradients off-palette
  • Hand-author shadows outside the light model
  • Use stage colors for status (or vice-versa)
  • Mix in a filled / duotone icon set
  • Add a second accent color anywhere
